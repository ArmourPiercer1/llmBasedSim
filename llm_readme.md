# 基于agent的互动模拟游戏

## Development Plan Initial
### 一、本工作区旨在实现一个 multi-agent 互动的模拟游戏
### 二、游戏架构需要包括：

#### （一）多agent系统：
1. 物理变化 agent：决定**非角色**的所有物理过程，并给出物理变化及它与人物交互的快过程结果。
2. 角色变化 agent：根据设定，决定角色的行为
3. 玩家感官 agent：决定玩家能接收到什么信息
4. 初始化 agent：通过与玩家对话，初始化工作流、角色设定、玩家设定等信息
例如，
`
角色变化agent A -> 角色A挥手 -> 物理变化agent -> 角色A站在桌子旁边，所以他会碰到桌子；桌子上有个花瓶，花瓶掉落 -> 玩家感官 agent -> 玩家看到花瓶掉落
`

#### （二）数值维护系统
存储玩家、各角色、物理世界的各种信息，以供agent作出决策

## Development Plan 20260623 -- Idea
### 一、角色扮演功能强化


#### （一）玩家意识-潜意识转换（精神层）

玩家的操作决策实际上是玩家的表层意识，但玩家的潜意识也会对行动造成影响。因此，“玩家输入的”和“世界中的玩家想要做的”会有所区别。这就需要一个潜意识节点对玩家的输入操作进行处理。例如，
```
玩家的人设是一个傲娇大小姐。虽然她喜欢一个npc，但她就是做不到直接说。
若玩家输入“我对{喜欢的对象}说‘我爱你。’”，潜意识节点需要将其处理为“哼，才没有喜欢你呢！”
```
但潜意识处理层也是需要更新的。比如，
```
傲娇大小姐经历了自己喜欢的人被和自己的情敌约会、并明确表达出好感的事情后，痛改前非。
现在，她能正常说出“我爱你”了。
```
实现这一功能至少需要维护玩家角色的潜意识记录，及负责潜意识处理的节点。玩家的输入应当在这一层被处理为结构化输出。

#### （二）玩家想法-行动转换（物理层）：

玩家在进行操作决策时，实际上输入的是“玩家接下来想进行的行动/说的话”，被潜意识层翻译后也是如此。但现实世界中，从“玩家想要进行的”到“玩家实际进行的”之间有着巨大鸿沟。例如，玩家可能输入“我直接飞起来，击杀了大boss”，但这在大多数场景下是不可行的。

为获得沉浸式体验，需要针对玩家的操作决策进行转换，去除其中不可行的部分。你需要加入必要的节点，并维护“玩家能做什么”的数据库。例如，新增一个`User Capibility List`数据库，并在其中维护
```
1. 符合对于普通人认知，但玩家不能做的事情
    例如，若玩家的设定是一个没有腿的残疾人，那么他就不能走路；
2. 不符合普通人认知，但玩家能做的事情
    例如，玩家人设是一个修仙者且修为足够高，玩家就真的可以飞起来并直接杀死大boss
3. 玩家能够概率成功的事情
    例如，玩家是一个不太熟练的盗贼，他的开锁术等级不够高，打开一些锁时就需要roll点
```
实现这一功能至少需要维护玩家角色的能力数据，并提示物理引擎节点进行处理。

### 二、游戏工作流优化
#### （一）从记录角色和世界信息的文件启动游戏
应当允许玩家通过结构化的文件直接输入`玩家角色设定、npc设定、环境设定`等信息，跳过当前的初始化流程直接启动游戏。

#### （二）存档及恢复功能
在`从记录角色和世界信息的文件启动游戏`功能完成后，增加存档功能

#### （三）维护玩家、npc和环境的物理数据
实现：
1. 玩家和 npc 的物理数据。例如，身高和三围，进而处理`玩家是个大胖子 -> 进不去一个小门`的推理。
2. 环境的部分物理数据。例如，某个门的尺寸，进而处理前述`玩家是个大胖子 -> 进不去一个小门`的推理。

#### （四）语言表达范例
允许在设定玩家和npc角色时，提供一些“之前这个角色是怎么说话的”信息，以及一些示例语段，将其注入到对应的处理节点提示词中，以维护人设和文风的一致性
